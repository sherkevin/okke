## 0. 先给出最终决策

这次任务分配不再以“把所有链条都跑到看起来完整”为目标，而是以**论文可写性和证据可信度**为目标。

当前正式判断如下：

1. **`TLRA_zero` 降级为 viability probe / appendix 身份。**
   - 依据：`Stage 0` 图像级 overlap 很高，但 patch-level overlap 偏低。
   - 结论：不要再把 `TLRA_zero` 当主表核心方法去包装。

2. **`VASM` 保留为主线必要模块。**
   - 依据：`MMBench / MME / MMMU pilot` 都显示 `tlra_full > tlra_no_vasm`。
   - 结论：当前最强正证据来自 `Chain B`，应优先固化。

3. **`AdaptiveTopK > MeanPool` 当前不能强写。**
   - 依据：`FREAK` 当前是 `MeanPool 0.235 > AdaptiveTopK 0.225`。
   - 结论：`Chain C` 现在是风险链，而不是胜利链。

4. **`DocVQA` 的角色改为负控制 / OCR concession 复核。**
   - 依据：数据已到位，但结果没有形成正证据。
   - 结论：不要再把它当“主正结果数据集”，而是当 safety boundary。

5. **视频继续停留在 appendix-only / bounded pilot。**
   - 依据：`VidHalluc` 为空结果，`Video-MME` 仍未达到可评测布局。
   - 结论：现在不允许视频反向主导论文主合同。

6. **最优先的不是“再找新故事”，而是把主线写成可交付结果。**
   - 主线顺序：
     - `Chain B` 正式化
     - `Chain A` 补完整
     - `DocVQA` 作为负控制跑通
     - `Chain C` 再决定是救还是降级

## 1. 两位工程师的职责边界

### 工程师 A：主线正证据负责人

你的目标是：**把论文当前最可信的正证据做厚、做稳、做成正式主表。**

你负责：

- `Stage 0` 的最终定位与 `TLRA_calib` 资产冻结
- `Chain A` 的正式主表
- `Chain B` 的正式主表
- `MMMU Hard` 的冻结

你**不负责**：

- `DocVQA` 结果修通
- `VidHalluc / Video-MME` 工程排障
- 为 `AdaptiveTopK` 强行调参找赢面

### 工程师 B：风险链与负控制负责人

你的目标是：**把当前最危险、最容易让论文失真的链条做成“诚实可写”的结果。**

你负责：

- `DocVQA` 负控制正式跑通
- `FREAK` 复核与 `MeanPool / AdaptiveTopK / RandomK` 风险判断
- `VidHalluc` 最小可读结果
- `Video-MME` 是否可评测的布局结论

你**不负责**：

- `MMBench / MME / MMMU Hard` 主表
- 继续扩写 `TLRA_zero` 主方法身份
- 为视频追加主表级叙事

## 2. 工程师 A 的任务

## A1. 冻结 `TLRA_calib` 正式 checkpoint 身份

### 目标

确认 `Phi_calib` 的正式路径、来源、训练方式、适用模型。

### 交付物

- 一个正式 checkpoint 路径
- 一份 provenance 说明
- 一份调用命令样例

### 成功标准

- 日志和文档里不再出现 `REPLACE_ME.pt`
- 之后所有 `TLRA_calib` 结果都指向同一个 checkpoint

### 失败标准

- 如果 2 小时内仍无法冻结，就直接出结论：
  - `TLRA_calib` 暂不进入正式主表
  - 当前论文继续以 `TLRA_zero probe + TLRA contract` 的方式保守书写

## A2. 补完整 `Chain A` 主表

## 3. 工程师 B 的任务

## B1. 把 `DocVQA` 固定为负控制结果，而不是“待下载任务”

### 目标

现在不再追求 `DocVQA` 上的正向提升，而是追求**负控制可写**。

### 必做内容

- 修通 `DocVQA` 正式可读结果
- 输出：
  - `trigger_rate`
  - `AGL`
  - 主指标（即使接近 0 也要诚实记录）
  - 与 `Base` 的差异

### 成功标准

- 形成一份“`DocVQA` 近零触发 / 近零收益 / 近零伤害”的可引用负控制结果

### 失败标准

- 如果仍只能得到空结果或 `Loaded 0 samples`
  - 必须交付 loader/layout 诊断
  - 明确指出是代码、路径、标注格式还是数据布局问题

### 当前写作判断

- `DocVQA` 不再承担“证明方法有效”的任务
- 它承担的是“证明 OCR concession 是真的”的任务

## B2. 重做 `Chain C` 风险判断

### 目标

不要再默认 `AdaptiveTopK` 是胜利者，而是用最小但严谨的复核把这件事判死或判活。

### 必跑方法

- `TLRA_MeanPool`
- `TLRA_AdaptiveTopK`
- `TLRA_RandomK`

### 必跑数据

- `FREAK`

### 强要求

- 三个方法必须共享同一 `Phi_calib` 资产
- 不允许通过换资产、换数据、换 prompt 偷换 parity

### 结果解释规则

- 如果 `AdaptiveTopK` 仍不优于 `MeanPool`
  - 直接出结论：当前版本不支持强 local-evidence superiority claim
  - 论文中把这条 claim 降级为“待进一步验证”
- 如果 `AdaptiveTopK > MeanPool`，但差距极小
  - 仍然不能写强，除非统计上稳健

### 交付物

- `FREAK` 三方法统一批次结果
- 一份表 3 草稿
- 一段最终判断：
  - `AdaptiveTopK` 是否存活
  - `RandomK` 是否说明 gain 只是扰动效应

## B3. 视频只做 appendix-level 生死判断

### 目标

不要为视频主线烧预算，只做最小可读结论：

- `VidHalluc` 能否形成结果
- `Video-MME` 能否达到可评测布局

### 必做内容

- `VidHalluc` 至少拿到一份可读 JSON
- `Video-MME` 给出清晰结论：
  - 可评测
  - 不可评测，原因是什么

### 成功标准

- 至少有一个视频 pilot 可读
- 或者给出足够明确的 appendix-only / 暂不使用判断

### 失败标准

- 不允许继续停留在“目录存在但结果为空”这种中间态而不解释

### 当前写作判断

- 视频不影响主论文是否成立
- 视频只影响 appendix 里是否保留 exploratory pilot

## 4. 两位工程师共同遵守的规则
