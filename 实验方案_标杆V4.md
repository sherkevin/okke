## 1. 这版方案解决什么问题

`标杆V4` 的目标，不是发明新理论，而是把 benchmark 从“强协议草稿”推进成“真正可投稿的无数据论文模板”。

这意味着它必须同时做到四件事：

1. 论文正文里所有真正需要出现的主表、主图、附录图表都已经有占位。
2. 每一个主表/主图都能映射到明确实验，而不是停留在口头计划。
3. 对应实验要区分：
   - 已有结果
   - 已启动但未闭合
   - 尚未启动
   - 被代码缺口阻塞
4. 4 张 GPU 的分配必须服务于最短时间收回最关键论文资产，而不是平均分摊算力。

## 2. 核心结论：当前 benchmark 是否已经完全正确

结论：**还没有。**

当前 `标杆V3` 的理论主合同基本正确，但它还不是“真正论文 benchmark”的原因有四个：

1. 正文里虽然提到 `Table 1/2/3` 和 `Figure 1`，但还没有真正的可投稿占位骨架。
2. 附录虽然有 `Appendix A-G` 计划，但仍是清单式，而不是可直接填充的论文结构。
3. 一部分实验已经在跑或已有结果，但这个“运行状态”没有写进 benchmark 实验文件，导致工程与写作脱节。
4. `RandomK`、`MMMU Hard`、`Category Leakage Audit`、`suffix-collapse` 仍然是 benchmark 合同，但还不是稳定可执行代码路径。

因此，`标杆V4` 的价值是：**理论不重写，论文资产骨架补齐，实验状态显式化，GPU 调度与论文资产一一对应。**

## 5. 时间成本最优原则

在只有 4 张 5090、且论文接受性主要取决于主表质量而不是附录丰富度的前提下，最优策略不是“把所有任务均匀推进”，而是：

1. **先把能决定主论文是否成立的主表收回来。**
2. **对当前不支持强 claim 的链条，只跑到足以决定保留还是降级。**
3. **把代码缺口与实验长跑剥离，避免 GPU 空转等待工程修复。**

所以当前的合理并行不是“4 张卡同时盲跑”，而是“2 张卡跑确定主资产，1 张卡跑风险判定，1 张卡留给短任务/切换/补洞”。

## 9. 写作收缩规则

即使这些实验还没全部跑完，最新 benchmark 的写作边界也必须锁死：

1. `TLRA_zero` 默认写成 viability probe，除非新证据强到足以升格。
2. `VASM` 当前是主线最强资产。
3. `AdaptiveTopK` 不能强写优于 `MeanPool`，直到 `Table 3` 真正闭合。
4. `DocVQA` 只能写成 OCR concession 下的负控制。
5. `video` 只写 appendix-only bounded pilot。

## 10. 最终建议

`标杆V4` 已经把我们真正需要的论文资产骨架补齐了。后续的正确动作不是继续重写理论，而是：

1. 用 `GPU1 + GPU2` 收回 `Table 1 + Table 2` 的硬资产
2. 用 `GPU3` 决定 `Table 3` 是保留还是降级
3. 用 `GPU0` 做短任务、参数锁定和控制面
4. 用工程师3补齐 `RandomK / MMMU Hard / leakage / suffix / Video-MME loader`

只有这样，benchmark 才不是“很好看的计划书”，而是真正能支撑投稿的无数据论文模板。
