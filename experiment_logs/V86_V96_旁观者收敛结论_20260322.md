# V86-V96 旁观者收敛结论

日期：`2026-03-22`

## 1. 结论先行

结论：**现在不生成下一版 benchmark。**

理由不是“这轮没有价值”，而是相反：`V86-V96` 产生了大量高价值洞察，但这些洞察主要属于**方法定义校正、实验公平性收紧、失败边界量化**，还不足以支持一次新的 benchmark reset。

当前更合理的动作是：

1. 保留 `标杆V4` 作为当前 benchmark 主合同。
2. 把 `V86-V96` 的稳定洞察提炼成一个“收口清单”。
3. 等工程侧把 `Table 1`、合规 `DocVQA`、`MMMU Hard intervention_rate`、projector 身份等关键现实补齐后，再判断是否需要 `标杆V5`。

## 3. 跨轮稳定洞察

### A. 这些洞察应被保留

1. **`Base + LoRA` matched baseline 是必须项。**
   - 这是整个窗口最稳定、最高频、最有价值的 reviewer 共识。
   - 它不是可选补强，而是决定 `TLRA_calib` 是否具有独立方法价值的关键门槛。

2. **`VASM` 的工程现实必须写实。**
   - “offline precomputed vocab lookup + BPE continuation inheritance”
   - “无 online POS tagger in the loop”
   - 这类实现真实性是可信度资产，必须保留。

3. **`visual state provenance` 必须明确。**
   - `h_L^{(v_j)}` 从哪里来，是 final layer、mid layer、还是别的来源，不再是实现细节，而是方法定义本身。
   - reviewer 在多轮里反复把这一点打成核心问题，说明它必须进入 benchmark 级口径。

4. **`Top-M / Hijacking` 必须量化，而不是只写 limitation。**
   - `Hijacking CDF`
   - `Top-M ceiling`
   - 这些已经从“好看的 failure analysis”升级成方法边界的正式组成部分。

5. **公平性与边界比“大题眼”更值钱。**
   - 这轮高分点共同奖励的不是新叙事，而是更严格的 fairness boundary、更清晰的 failure boundary、更诚实的 evaluation contract。

### B. 这些洞察只应部分吸收

1. **mid-layer / layer sweep 是高价值问题，但还不该直接重写 benchmark 主合同。**
   - 它值得成为附加强审计或 appendix 方向。
   - 但当前工程现实还没有支持把它升格为下一版 benchmark 的主轴。

2. **continuous baseline 要更公平，这一点成立。**
   - 但从 `ContinuousAdd` 一路扩到 `Continuous_Gated_XAttn`，已经开始明显拉大主合同规模。
   - 可作为重要对照，不宜再无限扩展。

3. **MMHal / Distractor / multilingual audit 提醒了 context-blind 风险。**
   - 这些是高价值边界测试。
   - 但不能让论文主合同从“image grounding intervention”变成“context-blind noun booster 的不可能性研究”。

### C. 这些洞察不应升级为新 benchmark 主线

1. **不应把方法标题级自我定义升级成**：
   - `static late-fusion visual logit bias`
   - `bag-of-words noun booster`

2. **不应把 `VASM` 的最负面措辞升级成摘要级中心叙事。**
   - “fatal heuristic” 作为 reviewer 的攻击性用语有价值
   - 但不应直接变成 benchmark 自我定义

3. **不应同时把下列扩展线都并入主合同：**
   - CFG
   - MMHal
   - Distractor Prompt
   - multilingual audit
   - continuous-xattn
   - contrastive-calib-no-vasm
   - vocabulary-density sweep
   - high-res `N_v` scaling

判断：

- 这些扩展合在一起会让 benchmark 再次发散，且超过当前实验现实。

## 4. 与当前工程现实的匹配度

当前工程现实已经支持或部分支持的稳定洞察：

- `Base + LoRA` 脚手架已闭环
- `VASM` 实现说明已存在
- `visual_state_provenance` 已落入结果字段
- `Hijacking / Top-M / leakage / cost split` 的代码链路已开始成形

当前工程现实尚不支持直接升级 benchmark 的方向：

- 大规模 multilingual / tokenizer audit
- 复杂 continuous-xattn 正式基线
- 新训练目标下的 `Phi_calib` 正式对照
- 高分辨率 `N_v` scaling 全套实验
- 关系/负约束新主线全面并入正文

## 5. 为什么现在不出 `标杆V5`

因为 `V86-V96` 的主要价值是“**收紧旧 benchmark**”，不是“**产生一个新的更大 benchmark**”。

如果现在直接出 `标杆V5`，最容易犯三个错误：

1. 把 reviewer 的攻击性语言当成论文的正式自我定义。
2. 把尚未被实验资源支撑的大量扩展边界测试一股脑并入主合同。
3. 让 benchmark 从“可投稿模板”重新膨胀成“理论负面结果 + 无数风险实验”的大杂烩。

## 6. 更合理的下一步

下一次 benchmark 升级应满足至少两个条件后再进行：

1. `Table 1` 干净闭合。
2. 合规版 `DocVQA` 回收完成。
3. `MMMU Hard` 的 `intervention_rate` 解释清楚。
4. projector 身份冻结，`FREAK` 可从 provisional 升级。

在此之前，建议把 `V86-V96` 吸收成一个**benchmark 收口清单**，而不是直接生成 `标杆V5`。

## 7. Observer 最终判断

本窗口最重要的旁观者结论不是“我们发现了一个更新的大题眼”，而是：

- 论文真正该升级的是**方法定义的准确性**、**公平基线的严苛性**、**失败边界的量化方式**；
- 而不是继续把方法往一个过度负化、过度自我贬低、甚至接近自我否定的方向改写。

因此当前正式判断为：

**不生成下一版 benchmark；继续保留 `标杆V4` 为当前 benchmark，并把 `V86-V96` 当作一次高价值 observer 收口窗口。**
