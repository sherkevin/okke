## P0：必须优先修

### 1. `compare_bra_variants.py` 把多个 TLRA 方法错误折叠成同一路径

**问题：**

`compare_bra_variants.py` 当前把：

- `tlra_full`
- `tlra_adaptivetopk`
- `tlra_zero`

都映射到同一个 `bra_zero` 别名。

这会导致后续用这个脚本做对比、导表或复核时，方法语义被混掉，尤其会直接污染：

- `VASM` 必要性判断
- `AdaptiveTopK vs MeanPool` parity 复核

**建议修法：**

1. 重写 `normalize_method_name()`，不要再把 `tlra_full` 和 `tlra_adaptivetopk` 合并到 `bra_zero`。
2. 明确保留独立方法名：
   - `tlra_zero`
   - `tlra_full`
   - `tlra_adaptivetopk`
   - `tlra_meanpool`
   - `tlra_no_vasm`
   - `tlra_calib`
3. 导表和 CSV flatten 时保留原始 method 字段，不要做二次语义折叠。

**背景文件：**

- `compare_bra_variants.py`
- `论文大纲_标杆V3.md`
- `实验方案_标杆V3.md`

### 2. `TLRA_RandomK` 还停留在文档层，没有真正实现

**问题：**

`标杆V3` 和实验方案允许 `TLRA_RandomK` 作为强化对照，但当前代码执行路径里没有可直接运行的 `tlra_randomk`。

这意味着论文里提出的更强 parity control 目前不能被真实执行。

**建议修法：**

1. 在 `bra_logits_processor.py` 增加一个真正的 `random-k` patch selection 分支。
2. 它必须满足：
   - 使用与 `TLRA_AdaptiveTopK` 完全相同的 `Phi_calib`
   - 使用相同的 `k`
   - 只替换 patch 选择策略，其他逻辑完全不变
3. 在 `bra_eval_matrix.py` 的 `--bra_method` 里增加 `tlra_randomk`
4. 在 `run_eval_pipeline.py` 里按需增加 `tlra_randomk`
5. 在输出 JSON 中把该方法单独记录为 `tlra_randomk`

**背景文件：**

- `bra_logits_processor.py`
- `bra_eval_matrix.py`
- `run_eval_pipeline.py`
- `论文大纲_标杆V3.md`
- `工程师B_复制即跑命令_标杆V3_20260322.md`

### 3. `MMMU` 当前不是 `Hard`，甚至不是完整 MMMU，只是单一 `Art` 子集

**问题：**

`bra_eval_matrix.py` 里当前 `mmmu` 数据路径写死为：

- `MMMU_hf/Art/validation-00000-of-00001.parquet`

这说明现在的 `MMMU` 评测并不是：

- 完整 MMMU
- 也不是冻结后的 `MMMU Hard`

而只是单个子学科子集。

这会直接导致 `Chain B` 的论文口径失真。

**建议修法：**

1. 新建一个 `MMMU Hard manifest` 生成脚本，输出固定的 JSON/CSV manifest。
2. manifest 至少应包含：
   - `subject`
   - `parquet_path`
   - `sample_id`
   - `split_tag`
3. `bra_eval_matrix.py` 的 `load_mmmu()` 改为：
   - 支持多学科遍历
   - 支持按 manifest 加载
   - 支持 `--mmmu_manifest`
4. 在结果 JSON 里额外写出：
   - `mmmu_subjects`
   - `manifest_path`
   - `sample_count_by_subject`

**背景文件：**

- `bra_eval_matrix.py`
- `experiment_logs/v2_v7_task_audit_20260322.md`
- `实验方案_标杆V3.md`
- `双工程师并行任务_标杆V3_20260322.md`

### 4. `DocVQA` / `Video-MME` 的文本评测仍然过于粗糙

**问题：**

当前 `bra_eval_matrix.py` 中 `compute_text_match_metrics()` 只是做非常宽松的字符串包含判断：

- `pred == gt`
- `gt in pred`
- `pred in gt`

这对 `DocVQA` 和 `Video-MME` 都不够可靠：

- 对 `DocVQA`，至少应有更合理的 OCR/text normalization，最好支持 `ANLS`
- 对 `Video-MME`，当前这种子串匹配很容易虚高或失真

**建议修法：**

1. 为 `DocVQA` 增加规范化函数：
   - lowercase
   - trim
   - punctuation normalize
   - space normalize
2. 实现 `ANLS` 或最少实现近似文本相似度评分
3. 对 `Video-MME`：
   - 如果数据是开放答案，至少做 normalized exact match
   - 若存在选项题布局，则优先提取 letter answer
4. JSON 输出里明确区分：
   - `accuracy`
   - `normalized_exact_match`
   - `anls`（若实现）

**背景文件：**

- `bra_eval_matrix.py`
- `论文大纲_标杆V3.md`
- `实验方案_标杆V3.md`

### 5. `Video-MME` loader 仍然不适配当前真实数据布局

**问题：**

当前 `load_video_mme()` 只支持两种理想情况：

1. parquet 中直接有可访问的视频路径
2. json 中直接有可访问的视频路径

但远端实际状态是 `Video-MME` 仍偏向 chunk/zip 布局，这也是之前 `Loaded 0 samples` 的根本原因之一。

**建议修法：**

1. 单独写一个 `video_mme_index_builder.py`
2. 先把当前目录扫描成 index：
   - `video_id`
   - `resolved_video_path`
   - `source_chunk_or_zip`
3. `load_video_mme()` 改为优先读取这个 index，而不是临时猜路径
4. 如果数据仍是 zip/chunk，先做一次规范化展开或软链接整理
5. loader 失败时输出更明确的诊断：
   - `missing_annotation`
   - `missing_video_file`
   - `unresolved_video_id`
   - `bad_json_layout`

**背景文件：**

- `bra_eval_matrix.py`
- `工程师B_复制即跑命令_标杆V3_20260322.md`
- `experiment_logs/v2_v7_task_audit_20260322.md`

### 6. `paper_adversarial_workflow.py` 仍然对模型输出格式过于脆弱

**问题：**

旧工作流已经出现过因 section heading 不匹配而直接失败。当前 compact prompt 虽然更稳，但 workflow 仍然高度依赖：

- 精确 block markers
- 精确 section headings

只要模型轻微偏离，就会直接抛异常。

**建议修法：**

1. 给 `extract_block()` 增加更稳健的 fallback：
   - block marker 缺失时，尝试从 heading regex 恢复
2. 给 `run_scientist()` 增加 `retry with stricter formatting prompt`
3. 对失败输出自动保存 raw response，便于复盘
4. 将异常分为：
   - formatting failure
   - API failure
   - section splice failure
5. 若 compact 模式失败，可自动退回 full-paper revision 模式

**背景文件：**

- `paper_adversarial_workflow.py`
- `审稿工作流启动命令_V75.md`
- `论文大纲_V75.md`
